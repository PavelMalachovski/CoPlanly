## 2. Bugs (most severe first)

### 2.1 [High] Event repository mappers drop `sharedWith` / `permissions` / `lastModifiedBy`
`EventRepositoryImpl.toEntity()` and `toDomain()` do **not** map `sharedWith`(`sharedWithJson`), `permissions`, or `lastModifiedBy`.
- File: `app/src/main/java/com/coparently/app/data/repository/EventRepositoryImpl.kt` (`toDomain` ~L163, `toEntity` ~L190).
- Effect: every event that round-trips through Room loses its sharing list, permission level and last-editor. Because the editor loads the event from Room (already stripped) and then `copy()`-saves it, **editing an event can silently drop its shares** — the same class of data loss that #2.2 of the July audit fixed at the screen layer, re-introduced at the persistence layer. `toFirestoreMap()` still writes `sharedWith`, so the local DB and Firestore drift.
- Fix: add `sharedWith`, `permissions`, `lastModifiedBy` to both mappers (entity already has the columns).

### 2.2 [High / Config] Firestore rules not deployed → `PERMISSION_DENIED`
Reproduced live: opening the app logs `FirebaseFirestoreException: PERMISSION_DENIED` for `change_requests`. The deployed rules don't grant the new collections. Sync is now crash-guarded (#16), so it fails silently — but **change requests, and any cross-parent sync gated by those rules, do not work** until the rules are deployed.
- Action: `firebase deploy --only firestore:rules,storage`. `firestore.rules.simple` already covers `change_requests` / `expenses` / `notification_queue`; `storage.rules` covers `receipts/` + `event_images/`.
- Also: change-request / event push notifications require a Cloud Function to drain `notification_queue` — none is deployed, so no pushes are delivered.

### 2.3 [Medium] Multi-day / overnight events invisible on later days
The wired range query loads by start date only:
`SELECT * FROM events WHERE startDateTime >= :start AND startDateTime <= :end` (`EventDao.kt` ~L28).
A multi-day or overnight event that **starts before** the visible window is never loaded, so it doesn't appear on its later days (the month cell already filters multi-day correctly, but only among loaded events). A multi-day-aware query exists in the same DAO (~L40, `startDateTime <= :end AND (endDateTime IS NULL OR endDateTime >= :start)`) but isn't the path used. (Matches July audit §3.5.)
- Fix: use the overnight-aware query for the calendar range load.

### 2.4 [Medium] Production DB uses destructive migration fallback
`DatabaseModule.kt` (~L47–49) enables `fallbackToDestructiveMigrationFrom(...)`, `fallbackToDestructiveMigration()` and `...OnDowngrade()`. Any missing migration path or a downgrade **wipes all local data silently**. Acceptable in early dev; risky before the Play beta.
- Fix: keep explicit migrations only (remove the blanket destructive fallback) once the migration chain is complete.

### 2.5 [Medium] Strict `firestore.rules` reject the app's own writes
The strict rules validate an exact key set and `timestamp` types, but the app writes ~15 keys with ISO **string** dates, so every event create is denied. This is why the permissive `firestore.rules.simple` is deployed instead (see 2.2). (Matches July audit §2.3.)
- Fix: align the strict rules with the real document schema, test with the emulator, then deploy.

## 3. Improvements & polish

- **"Assigned To" selected state** (event editor): the selected "Dad" chip renders with a heavy blue border + drop shadow that doesn't match the "Mom" chip or the Material 3 look. Unify the selected-state styling for both parents.
- **New Event draft restore is silent**: opening "New Event" pre-fills the last unsaved draft (e.g. "Test fff") with no indication. Add a subtle "draft restored" hint (with an undo/clear), or only restore on an explicit prompt.
- **OAuth client secret ships in the APK** (July audit §2.1) — move to the Android OAuth client type (no secret) or proxy the token exchange through a Cloud Function, and rotate the secret.
- **`strings.xml` is gitignored** (July audit §2.1) — a fresh clone has no base string resources and can't build. Split real secrets into a separate ignored file and re-track ordinary strings.
- **`android:allowBackup="true"`** with sensitive family data (`AndroidManifest.xml` L18) — set `false` or provide backup rules that exclude the Room DB and preferences.
- **Deprecated `play-services-auth`** is still a dependency alongside Credential Manager (`build.gradle.kts` L165). Finish the migration and drop the legacy artifact.
- **No repository-layer tests** — `EventRepositoryImpl` (and the mapper bug in 2.1) would be caught by unit tests. Add them, starting with the event/expense/change-request repositories.
- **Empty states / pairing CTA**: Home ("Recent changes") and Weekly Summary only populate once paired. Add a clearer "pair with your co-parent" call-to-action / onboarding, since most value is gated on pairing.
