# CoParently — Full Project Audit (July 2026)

Audit date: 2026-07-19
Scope: architecture, code quality, security, data layer, tests, build & CI, roadmap compliance.
Baseline: `main` @ `9c65b33` (build passing).

Items marked **[FIXED]** were resolved during the audit session; the rest are recommendations.

---

## 1. Executive summary

The project is in good overall shape: Clean Architecture layering is real (domain/data/presentation),
Hilt DI is used consistently, the Compose UI is stateless with ViewModels as the source of truth,
and the build is green. The main risks found were: a **committed OAuth client secret**,
a **data-loss bug when editing shared events**, **Firestore rules that reject the app's own writes**,
an **API-34-only `java.time` call on minSdk 26**, and thin test coverage outside ViewModels.

MVP 1 (per `docs/CoPlanly/MVP_phases.md`, the authoritative roadmap) was completed during this
session — see §7.

## 2. Critical findings

### 2.1 OAuth client secret ships in the APK; string resources untracked — **ACTION REQUIRED**
The local `app/src/main/res/values/strings.xml` contains:

```xml
<string name="google_client_secret" translatable="false">GOCSPX-…</string>
```

Status clarification after checking git history: **all `strings.xml` files were removed from
tracking and gitignored** (commit `df718a1` "Add strings.xml files to .gitignore for
security"), and `git log -S GOCSPX` shows the secret value itself **never entered git
history**. So the repo is not leaking it — but two problems remain:

1. The secret still ships inside every APK (extractable by anyone with the APK).
   Move to the Android OAuth client type (no secret needed) or proxy the token exchange
   through a Cloud Function, then rotate the secret.
2. **A fresh clone of the repository cannot build**: gitignoring `strings.xml` removed
   *every* string resource from version control, so all `R.string.*` references fail to
   resolve (this is also why CI is red). Split real secrets into a separate ignored file
   (e.g. `secrets.xml`) and re-track the ordinary strings. New reminder strings were
   deliberately placed in tracked `reminders.xml` files as the first step of this pattern.

Note: `local.properties` and `app/google-services.json` are correctly gitignored.

### 2.2 Editing an event wiped its sync/sharing state — **[FIXED]**
`AddEditEventScreen` rebuilt the `Event` from scratch on save, resetting `sharedWith`,
`permissions`, `createdByFirebaseUid`, `isRecurring` and `createdAt` to defaults. Editing any
shared event silently unshared it. Now the loaded event is kept as a snapshot and `copy()` is
used, preserving all untouched fields.

### 2.3 Firestore security rules reject the app's own event writes
`firestore.rules` for `events.create` requires:
- `hasRequiredFields(...)` with an **exact** key set of `[title, startDateTime, endDateTime, eventType]` —
  but the app writes ~15 more keys, so the check always fails;
- `startDateTime is timestamp` — but the app writes ISO **strings**.

Under the strict rules every event create is denied; presumably the permissive
`firestore.rules.simple` is what is actually deployed (see `DEPLOY_RULES_NOW.md`), which
disables most protections.

**Recommendation:** align the rules with the real document schema (string dates, full key set,
`hasAll` in one direction only), test with the Firestore emulator, then deploy the strict rules.
Also: `notification_queue` allows unrestricted `create` for any authenticated user (spam vector).

### 2.4 `LocalDate.ofInstant` crashes below Android 14 — **[FIXED]**
`LocalDate.ofInstant()` exists only from API 34, while `minSdk = 26`. Date pickers would throw
`NoSuchMethodError` on Android 8–13. Replaced with
`Instant.atZone(...).toLocalDate()` (available from API 26) in all three call sites.

## 3. High-priority findings

### 3.1 `syncWithFirestore()` never completed — **[FIXED]**
`EventRepositoryImpl.syncWithFirestore()` called `.collect {}` on an infinite Room `Flow`,
so the coroutine suspended forever after the first emission. Replaced with `.first()`.

### 3.2 Massive duplication of the date-range computation — **[FIXED]**
The Month/Week/Day query-range calculation existed in **three** copies inside
`CalendarScreen` (initial load, pull-to-refresh, plus per-mode duplication). Extracted into a
single `queryRangeFor(viewMode, selectedDate)` function.

### 3.3 Firestore document map built in four places — **[FIXED]**
`EventRepositoryImpl` (3×) and `SyncService` each hand-built the event document map with
subtle differences (e.g. `sharedWith` was missing from repository writes — another way shares
were dropped). The repository now has one `toFirestoreMap()`; `SyncService` maps were aligned.

### 3.4 Dead code
- `CalendarDayContent` / `EventIndicatorDot` in `CalendarScreen` were unreferenced — **[FIXED]** (removed).
- `ViewModeSelector` is superseded by `ViewModeDropdown` but still compiled; consider deleting.
- `.cursorrules` line "Ты senior разработчик и понимаешь все аспекты разработки **игр**" is a
  leftover from another project.

### 3.5 Events spanning midnight are invisible on later days
`EventDao` range queries match only `startDateTime`. A multi-day event (or one crossing
midnight) disappears from every day except its first. Consider matching on
`startDateTime <= rangeEnd AND COALESCE(endDateTime, startDateTime) >= rangeStart`.

### 3.6 `observeEventsForParents` is not an observer
The method name promises real-time updates but performs a one-shot `get()` inside a `flow {}`.
Use `addSnapshotListener` + `callbackFlow` for actual real-time behaviour, or rename.

## 5. Architecture assessment

**Strengths**
- Clean separation: `domain` (models, repositories interfaces, use cases), `data` (Room, Firestore,
  sync), `presentation` (Compose + ViewModels). Dependencies point inward.
- Hilt modules are organized per concern (Database/Firebase/Google/UseCase/Security/AI).
- Offline-first: Room is the source of truth, Firestore is a sync target.
- WorkManager + HiltWorkerFactory correctly configured for background work.
- Extensive feature set beyond roadmap: chat, expenses, medical records, education, AI (Gemini).

**Weaknesses**
- Some domain use cases depend on `data.analytics`/`data.crashlytics` managers directly
  (an inward-pointing violation; acceptable pragmatism, but an interface would be cleaner —
  the new `ReminderScheduler` shows the intended pattern).
- `presentation/calendar` mixes helpers (`CustodyHelper`, `DateRangeHelper`) with views; fine
  for now, but they are pure logic and could move to `domain`.
- The 222-file codebase has grown multiple parallel "phases" of documentation in `docs/`
  describing the same screens; consolidated during this audit (see `docs/archive/`).
